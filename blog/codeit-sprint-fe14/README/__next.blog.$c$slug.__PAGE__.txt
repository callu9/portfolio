<h2>PART 1 (2024.12.30 ~ 2025.02.08)</h2>
<h3>1-1. 웹 퍼블리싱 시작하기 (HTML/CSS)</h3>
<p>flexbox, grid의 여러 속성을 많이 알지 못한다는 것을 깨달았습니다.</p>
<p>이를 위해 실습 자료와 <a href="https://github.com/callu9/14-Sprint-Mission">스프린트 미션</a>을 활용하여 HTML, CSS의 활용을 익히기 위해 힘썼습니다.</p>
<h3>1-2. Git 기초</h3>
<p>그동안 실무에서 사용했던 전략(main을 기반으로 develop 및 feature 브랜치를 통해 개발을 진행)이 Git Flow 전략이며, 이를 설명하는 대표적인 <a href="https://techblog.woowahan.com/2553/">기술 블로그</a>에 대한 이해도도 높일 수 있었습니다.</p>
<p>또한, Branch Merge 방법에 Rebase, Squash 등이 있음을 알 수 있었으며, 이를 추후 프로젝트를 진행함으로써 몸소 겪을 수 있었습니다.</p>
<h3>1-3. Javascript</h3>
<p>Javascript로 주로 코딩을 해왔으나 깊이 있는 지식이 부족하다고 깨달았습니다. 특히 자바스크립트의 비동기 방식이나, this, 실행 컨텍스트 등 이해도가 낮은 부분이 있어 이를 보충하기 위한 방법을 더 찾아야 했습니다.</p>
<p>멘토링 시간을 통해 <strong>모던 자바스크립트 Deep Dive</strong>를 함께 읽으며 이해를 높일 수 있었으며, 이해가 어렵거나 외우기 힘든 부분에 대해 스터디 방식 피드백을 받기도 하였습니다.</p>
<h2>PART 2 (2025.02.10 ~ 2025.03.29)</h2>
<h3>2-1. React</h3>
<p>레슨을 통해 쉽게 시작했던 리액트 프로젝트의 렌더링 방식 및 Hook 사용에 대한 지식을 채우고 원리를 이해하여 개발하고자 하는 자세를 잡을 수 있었습니다.</p>
<h3>2-2. Cookie, Session Storage, Local Storage</h3>
<p>지식보다는 활용 경험만 있던 저에게 지식을 채워줄 수 있는 과정이었습니다.</p>
<h3>2-3. 기초 프로젝트 : <a href="https://github.com/codeit-fe-14-first-project-team2/fandom-k">Fandom-K</a></h3>
<p>깃허브 레포지토리 생성, 커밋/네이밍 컨벤션 정의, 공통 스타일 정의를 통해 구조를 잡고 개발 리딩을 하였습니다.
가이드를 위해 리드미 문서의 정리를 하였고, 이를 참고하여 개발할 수 있도록 전달하였습니다.</p>
<p>또한 팀원들이 빠르고 편한 개발을 진행할 수 있게 공통 컴포넌트의 개발을 우선 진행한 후, 메인 랜딩 페이지와 추가 기능의 개발을 진행했습니다.</p>
<p>궁금한 점이나 어려운 점을 충분히 공유하지 못해 각자 어떤 점에서 헤매고 있는지 파악하기 어려웠습니다.</p>
<p>하지만, 초반 구조 설계와 컴포넌트 설계 및 개발을 잘 진행했다는 멘토님의 피드백, 처음 구조를 잘 잡아 시작하기 편하였다는 팀원들의 피드백을 받았고,
일정 내 프로젝트 기능 개발과 배포, 발표자료 작성까지 잘 끝낼 수 있었습니다.</p>

<h3>3-3. 중급 프로젝트 : <a href="https://github.com/callu9?tab=repositories#:~:text=FE14%2DTeam7%2DTaskify/Taskify">Taskify</a></h3>
<p>공통 디자인 시스템 정의 및 반영, 공통 컴포넌트 개발은 모두에게 까다로운 작업이었고, 이로 인해 개발 일정 지연에 이르게 되었습니다.</p>
<p>저는 제가 힘 닿는 곳까지 개발을 진행하고자 하여, 개발 일정이 지연된 팀원들을 위해
먼저 2차 업무 분배를 진행하거나, 남은 업무나 까다로운 업무를 자발적으로 맡아 해나가기 시작했습니다.
제가 팀원들 중 UI 구현이 가장 빠르다고 판단했기 때문입니다.</p>
<p>UI 개발을 놓친 부분 없이 발표 세션 날을 맞이할 수 있었습니다.
그러나, UI 개발이 완전히 피그마와 동일하지 않은 화면을 발견했고, 사전에 팀원의 동의를 받고 스타일 변경 및 코드 수정에 나섰습니다. 배포 전 화면 스타일링 및 기능 구현이 우선이었던 저와, 개발 경험을 우선으로 했던 팀원의 우선순위 차이가 있었던 점에서 팀 프로젝트에 있어 공통적인 우선순위 협의와 공유가 중요한 것을 깨달았습니다.</p>
<p>또한 어떤 공통 컴포넌트를 써야하는지, 어떻게 쓸 수 있는지 혼란이 있었던 점은 깃허브 위키를 통해 정보를 공유할 수 있다는 점을 발표 세션을 통해 깨달았습니다. 인증 정보 저장을 위한 Local Storage 사용을 자제하라는 점과 SSR 사용이 저조하다는 강사님 피드백을 받고 개선점을 찾아나갈 수 있었습니다.</p>

<h3>4-2. 고급 프로젝트 : <a href="https://github.com/FE14-part4-team4-globalnomad/GlobalNomad">GlobalNomad</a></h3>
<p>유저가 판매자이자 체험 참여자로서 체험을 등록/관리하고 내 예약을 관리할 수 있는 서비스를 개발합니다. 유저가 로그인/회원가입/카카오 간편인증 할 수 있도록 하는 인증 관련 기능 및 프로필 수정 기능을 담당하였습니다.</p>
<p>유독 어려워하는 부분이었던 인증 관련을, 관련 기능 구현 경험이 있던 제가 맡음으로써 빠른 처리를 위해 노력했습니다.
또한 연관 있는 프로필 수정 기능을 담당하였으며, 원활하고 빠른 팀 활동 진행을 위해 시간이 남을 경우 남는 업무를 담당하는 등 열정적으로 임했습니다.</p>
<p>처음 사용해보는 zustand, persist 라이브러리를 사용해보면서 기존 localStorage를 통해 유저 정보 및 토큰을 관리하던 습관을 버리고 새로운 기술에 적응하기 위해 노력하였습니다.
이에 어려움을 많이 느끼기도 했지만, 새로운 기술에 적응하고 더 나은 기능 구현을 위해 리팩토링을 해나가는 과정이 성장하고 있다는 뿌듯함을 가져다 주었습니다.</p>
<p>또한 빠른 진행을 위해 남은 업무를 담당하고, 이후 서비스 기획의 부족한 부분 혹은 UX 개선 가능성 있는 부분을 개선하기 위해 노력하고자 했으나
커스텀 캘린더로써 예약현황 기능을 구현할 때 애를 먹어 계획보다 일정이 지연되기도 하여 고민이 컸습니다. 우선 팀 협의 사항으로 "기능 구현"에 우선을 두었기에
기획 개선보다는 기능의 완성을 우선으로 두고 더 나은 기능 구현을 위해 노력했습니다.</p>
<p>useState만 사용하던 기능을 useReducer를 사용해 DX를 개선하고, 용어 및 사이드메뉴 이름을 변경함으로써 UX를 개선하여 사용성을 높이는
노력을 프로젝트 기간 내 하지 못한 점은 많이 아쉬웠습니다.
그러나, 이를 기반으로 더욱더 개선해나가며 성장할 수 있는 기회를 얻었음에 뿌듯함과 열정을 이어가고자 합니다.</p>
