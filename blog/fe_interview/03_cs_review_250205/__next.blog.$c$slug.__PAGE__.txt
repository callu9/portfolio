<h2>Closure</h2>
<p>중첩 함수는 이미 생명 주기가 종료한 외부 함수의 변수를 참조할 수 있다. 이러한 중첩 함수를 <strong>클로저</strong>(closure)라고 한다. 클로저는 중첩 함수가 상위 스코프의 식별자를 참조하고 있고 중첩 함수가 외부 함수보다 더 오래 유지되는 경우에 한정하는 것이 일반적이다.</p>
<ul>
<li>자유 변수 (free variable): 클로저에 의해 참조되는 상위 스코프의 변수</li>
<li>클로저 (closure): 함수가 자유 변수에 대해 닫혀있다. 즉, <strong>자유 변수에 묶여있는 함수</strong>다.</li>
<li>[참고] 모던 자바스크립트 Deep Dive: <a href="../javascript-deep-dive/24_closure.md">24. 클로저</a></li>
</ul>
<pre><code class="language-js">const x = 1;

function outer() {
  const x = 10;
  const inner = function () { console.log(x); }; 
  return inner;
}

const innerFunc = outer();
innerFunc(); // 10
</code></pre>
<h3>Closure 사용 이유</h3>
<ul>
<li>상태를 안전하게 변경하고 유지하기 위해 사용</li>
<li>즉, 상태가 의도치 않게 변경되지 않도록 안전하게 은닉하고, 특정 함수에게만 상태 변경을 허용하기 위해 사용</li>
</ul>
<h3>Closure 사용 시기</h3>
<ul>
<li>부수 효과를 최대한 억제하여 오류를 피하고 프로그램의 안정성을 높이기 위해 클로저가 적극적으로 사용된다.</li>
<li>캡슐화: 객체의 상태를 나타내는 프로퍼티와 프로퍼티를 참조하고 조작할 수 있는 동작인 메서드를 하나로 묶는 것. 정보 은닉의 목적으로 사용하기도 한다.</li>
<li>정보 은닉: 객체의 특정 프로퍼티나 메서드를 감추는 것.
<ul>
<li>외부의 접근으로부터 객체의 상태가 변경되는 것을 방지해 정보를 보호하는 효과</li>
<li>객체 간의 상호 의존성을 낮추는 효과</li>
</ul>
</li>
</ul>
<h2>Event Bubbling</h2>
<ul>
<li>이벤트 버블링: 자식 → 부모, <code>e.stopPropagation()</code> 사용을 통한 전파 방지 가능</li>
<li>이벤트 캡처링: 부모 → 자식</li>
<li>이벤트 위임: Bubbling 이용한 기법으로, 하위 요소에 각 이벤트 등록할 필요 없이 상위 요소에 이벤트를 등록하여 하위 요소 이벤트를 발생시키는 기법</li>
</ul>
<h2>Microtask Queue</h2>
<ul>
<li>Event Loop: <strong>브라우저 동작 타이밍을 제어하는 관리자 역할.</strong> 이벤트 루프는 태스크가 들어오길 기다렸다가, 태스크가 들어오면 이를 처리하고, 처리할 태스크가 없는 경우엔 잠드는, 끊임없이 돌아가는 자바스크립트 내 루프
<ul>
<li>참고: https://youtu.be/8aGhZQkoFbQ</li>
</ul>
</li>
<li>Callback Queue (Task Queue or Microtask Queue) → Call Stack → Web APIs</li>
</ul>
<h3>Microtask Queue vs. Task Queue</h3>
<ol>
<li>
<p>Task Queue</p>
<ul>
<li>스크립트 실행, 이벤트, HTML 파싱, 콜백, Fetch, Ajax, DOM 조작, <strong>setTimeout</strong> 등</li>
<li>위 작업들은 task queue에 쌓여 있다가 이벤트 루프에 의해 stack이 비어 있을 때 stack으로 올라가게 됩니다.</li>
</ul>
</li>
<li>
<p>Microtask Queue</p>
<ul>
<li>microtask queue(job queue)라는 별개의 queue에 쌓였다가 이벤트 루프에 의해 call stack으로 올라가게 됩니다.</li>
<li>이 microtask에 해당하는 것 중 하나가 바로 <strong>Promise</strong>입니다.</li>
<li>(그 외에도 process.nextTick, Object.observe, MutationObserver가 있다고 합니다.)</li>
<li>microtask가 task보다 <strong>높은 우선 순위</strong>를 가지고 있습니다.</li>
</ul>
</li>
</ol>

<h3>FSD (Feature-Sliced Design)</h3>
<p>Feature-Sliced Design의 주요 특징 중 하나는 <strong>계층적 구조</strong> (hierarchical structure)를 가진다는 점이다. 이는 한 방향으로만 흐르는 선형 플로우(linear flow)를 유지하기 위한 목적을 가지고 있다. 상위 계층의 레이어는 자신보다 낮은 계층의 레이어만 사용할 수 있다.</p>
<ul>
<li><strong>app</strong>: 앱의 로직이 초기화되는 곳이다. Provider, 라우터, 전역 스타일, 전역 타입 선언 등이 정의된다. 앱의 진입 포인트의 역할을 한다.</li>
<li>processes: 멀티 스탭 가입과 같이 여러 페이지에서 사용되는 프로세스를 관리하는 레이어이다. 이 레이어는 deprecated 된 것으로 여겨지긴 하지만, 간혹 반영되어있는 걸 볼 수 있다. Processes 또한 옵셔널 레이어이다.</li>
<li><strong>pages</strong>: 앱의 페이지들을 포함하고 있는 레이어이다.</li>
<li><strong>widgets</strong>: 페이지들에서 사용되는 standalone UI 컴포넌트들이 있는 곳이다.</li>
<li><strong>features</strong>: 비즈니스 가치를 지닌 유저 시나리오와 기능에 관한 것들이 포함된 레이어이다. 예를 들어, 좋아요, 리뷰 작성, 상품 별점 주기 등이 여기에 포함된다. 여기 또한 옵셔널 레이어이다.</li>
<li>entities: 유저, 리뷰, 댓글 등과 같은 비즈니스 개체(entities)를 뜻한다. 옵셔널 레이어이다.</li>
<li><strong>shared</strong>: 특정 비즈니스 로직을 위한 것만이 아닌 범용적으로 사용되는 재사용가능한 컴포넌트와 유틸리티를 포함하는 레이어이다. UI 키트, axios 설정, 앱 설정, 또는 비즈니스 로직에 묶여있지 않은 각종 핼퍼 등이 여기에 포함된다.</li>
</ul>
<p>위 7개의 각각의 레이어는 각자의 책임 영역이 있으며, 비즈니스를 기반으로 한다. 위와 같은 레이어 정리는 코드베이스를 정리하고 유지보수성과 확장성 높은 모듈 기반의 구조를 만드는데 도움을 준다.</p>
<p>FSD의 주요 목적 중 하나는 <strong>높은 응집도와 낮은 결합도</strong>(high cohesion and loose coupling)를 달성하는 것이다. OOP에선, 이러한 문제들을 다형성(polymorphism), 캡슐화(encapsulation), 상속(inheritance), 그리고 추상화(abstraction)와 같은 컨셉들로 해결한다. 이러한 개념들은 코드의 고립화와 재사용성, 그리고 다재다능성(versatility)을 보장해주며, 이를 통해 컴포넌트 또는 기능(functionality)이 어떻게 사용되느냐에 다른 결과가 만들어질 수 있도록 해준다. FSD는 이러한 원칙들을 프론트엔드에서 적용할 수 있도록 도와준다.</p>
<p>추상화와 다형성은 <strong>레이어</strong>를 통해 얻을 수 있다. 하위 계층에 있는 레이어들은 추상적이기 때문에, 높은 레이어들에서 재사용할 수 있다. 또한, 조건에 따라, 컴포넌트 또는 기능은 특정 파라미터 또는 props에 따라 다른 기능을 보여줄 수도 있다.</p>
<p>캡슐화는 Public API의 사용을 통해 보장받는다. Public API는 슬라이스와 세그먼트 외부에서 불필요한 기능들을 고립시킨다. 슬라이스의 내부 세그먼트로의 접근은 제한되며, Public API를 통해서만 슬라이스 또는 세그먼트의 기능과 컴포넌트에 접근할 수 있다.</p>
<p>상속 또한 레이어를 통해 보장된다. 상위 계층에 있는 레이어들이 하위 게층들을 재사용할 수 있기 때문이다.</p>

<li>구조의 표준화</li>
<li>확장성</li>
<li>개발 스택으로부터 독립적인 방법론</li>
<li>모듈간 기대하지 못한 사이드 이펙트 없는 컨트롤되면서 명확한 연결</li>
<li>비즈니스 기반 구조적 방법론</li>
</ul>

<li>컨셉에 대한 이해, 팀 문화, 컨셉 충실도 필요</li>
<li>이슈 발생 시 즉각적인 대응이 요구됨. 코드 문제와 컨셉으로부터 벗어난 코드가 즉각적으로 눈에 들어오게 됨. 그러나, 사실 이건 장점으로 볼 수도 있음.</li>
</ul>
<h2>React Query</h2>
<p><strong>서버 상태 관리</strong>를 쉽게 할 수 있게 도와주는 라이브러리</p>
<ul>
<li>서버 상태 관리: 서버에 요청하고 응답받는 모든 과정과 연관된 데이터들
<ul>
<li>fetching: 서버에서 데이터 받아오기</li>
<li>caching: 서버에서 받아온 데이터를 따로 보관해서 동일한 데이터가 단 시간 내에 다시 필요할 시 서버 요청없이 보관된 데이터에서 꺼내쓰기</li>
<li>synchronizing: 서버 상의 데이터와 보관 중인 캐시 데이터(서버 상태)를 동일하게 만들기(동기화)</li>
<li>updating: 서버 데이터 변경 용이 (mutation &#x26; invalidateQueries)</li>
</ul>
</li>
</ul>
<h3>기존 미들웨어의 한계</h3>
<p>서버와의 API통신과 비동기 데이터 관리를 위해 Redux-thunk, Redux-Saga등의 미들웨어를 사용할 수 있다. 하지만 아래의 문제점이 있다.</p>
<ul>
<li>보일러 플레이트 코드의 양이 너무 많다.</li>
<li>Redux는 비동기 데이터 관리를 위한 전문 라이브러리가 아니다. (규격화 문제)</li>
</ul>
<h3>리액트 쿼리의 장점</h3>
<ul>
<li>보일러 플레이트를 만들다가 오류날 일이 없다</li>
<li>사용방법이 thunk에 비해 쉽고 직관적이다.</li>
</ul>
<h3>기존 미들웨어와의 차이</h3>
<pre><code class="language-js">// React Query 미사용 시
const [todos, setTodos] = useState([]);
const [isLoading, setIsLoading] = useState(false);

const getTodos = async () => {
  setIsLoading(true);
	const data = await axios.get(`${API_URL}/todos`).then(res => res.data);
	setTodos(data);
  setIsLoading(false);
}
useEffect(() => {
	getTodos();
}, []);


// React Query 사용 시
const getTodos = () => axios.get(`${API_URL}/todos`).then(res => res.data);

const { data: todos, isLoading } = useQuery(["todos"], getTodos);
</code></pre>
<ul>
<li>[참고] <a href="https://tech.kakaopay.com/post/react-query-1/#-%EB%84%88%EB%AC%B4-%EC%9E%A5%ED%99%A9%ED%95%9C-boilerplate-%EC%BD%94%EB%93%9C">카카오페이 프론트엔드 개발자들이 React Query를 선택한 이유</a></li>
</ul>
<h3>주요 키워드</h3>
<h4>Query</h4>
<ul>
<li>어떤 데이터에 대한 요청</li>
<li>axios의 get요청과 유사</li>
</ul>
<h4>Mutation</h4>
<ul>
<li>데이터 그룹 자체를 변경하는 것 (Create, Update, Delete)</li>
<li>axios의 post, put, patch, delete 요청과 유사</li>
</ul>
<h4>Query Invalidation</h4>
<ul>
<li>쿼리 무효화</li>
<li>기존에 가져온 Query는 서버 데이터이기 때문에 언제든지 변경이 있을 수 있다. 최신 상태가 아닐 수 있으므로 기존의 쿼리를 무효화한 후 최신화 해야 하는데 React Query는 이런 과정을 알아서 해준다.</li>
</ul>
<h3>SWR 전략</h3>
<ul>
<li>SWR (Stale-While-Revalidate) 전략: 신규 데이터가 도착하는 동안 일단 기존 캐싱된 데이터를 사용하도록 하는 전략</li>
<li>서버의 헤더 응답 설정 Cache-control에서 아이디어 기원</li>
<li>클라이언트가 0 ~ 1s 사이에 다시 데이터를 요청하면, 서버 호출없이 캐시 데이터를 바로 사용</li>
<li>클라이언트가 1 ~ 60s 사이에 다시 데이터 요청하면, 일단 캐시 데이터를 사용하고 서버에서 신규데이터를 주면 그것으로 교체</li>
</ul>
<h3>캐시 데이터 저장</h3>
<ul>
<li>QueryClientProvider는 React Context API를 내부적으로 사용한다.</li>
<li>QueryCLientProvider의 자식으로 있는 모든 컴포넌트들은 캐시 데이터에 접근할 수 있다.</li>
<li>페이지 컴포넌트 외부에 상태가 존재한다는 점에서 캐시 데이터는 전역 상태로 볼 수 있다.</li>
</ul>
<pre><code class="language-js">//App.jsx
const queryClient = new QueryClient();

const App = () => {
  	&#x3C;QueryClientProvider client = {queryClient}>
      	&#x3C;Router/>
  	&#x3C;/QueryClientProvider>
	};
}
</code></pre>
<h3>useQuery에서 자주 사용하는 옵션들</h3>
<p>① enabled</p>
<pre><code class="language-js">useQuery(["todos"], getTodos, {enable: true})
</code></pre>
<ul>
<li>boolean 타입 (true / false)</li>
<li>true일 경우에만 queryFn 실행</li>
<li>default 값은 true, useQuery 자동 실행됨</li>
</ul>
<h4>예제1 (Disabling / Pausing Queries): 이벤트 발생 시에만 수동 실행하고 싶을 때</h4>
<pre><code class="language-js">const {data, refetch} = useQuery(["todos"], getTodos, {enabled: false});

return (
	&#x3C;div>
  		&#x3C;button onClick = {() => refetch()}>
  			데이터 불러오기
        &#x3C;/button>
	&#x3C;/div>
);
</code></pre>
<h4>예제2 (Dependent Queries): useQuery가 2개 이상이고 실행순서 설정이 필요할 때</h4>
<pre><code class="language-js">//Get user
const {data: user} = useQuery({queryKey: ['user', email], queryFn: getUserByEmail});

const userId = user?.id;

//Get user's projects
const { status, fetchStatus, data: projects }
	= useQuery({ queryKey: ['projects: userId'], queryFn: getProjectsByUser, enabled: !!userId});
// userId가 존재하기 전까지 이 쿼리는 실행되지 않음
// !!userId === Boolean(userId)
</code></pre>
<p>② select</p>
<ul>
<li>queryFn에 의해 리턴된 값을 변형시킨 후에 useQuery의 리턴 data로 넘겨줌
(단, cache data는 queryFn에서 리턴 받은 값 그대로이다.)</li>
</ul>
<pre><code class="language-js">import {useQuery} from "react-query"

function User(){
  const { data } = useQuery( ['user'], fetchUser, { select: (user) => user.username, });
  return &#x3C;div>Username: {data}&#x3C;/div>;
}
</code></pre>
