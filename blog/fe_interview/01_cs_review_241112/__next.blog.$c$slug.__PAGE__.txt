2:T1dfa,<h2>React Hook</h2>
<p>Q. 리액트 훅을 사용할 때 필수 조건은 무엇인지 나열하고 설명하시오.</p>
<ol>
<li><strong>훅은 최상위에서만 호출해야 한다.</strong>
<ul>
<li>훅은 루프, 조건문, 중첩 함수 안에서 호출하면 안 됩니다. 이를 위반하면 상태가 꼬이거나 올바르게 관리되지 않습니다.</li>
</ul>
</li>
<li><strong>훅은 리액트 함수 컴포넌트나 커스텀 훅 내부에서만 호출해야 한다.</strong>
<ul>
<li>일반 함수나 클래스 컴포넌트에서는 훅을 사용할 수 없습니다.</li>
</ul>
</li>
<li><strong>useEffect 또는 useCallback의 의존성 배열 관리에 주의해야 한다.</strong>
<ul>
<li>의존성 배열을 올바르게 작성하지 않으면, 원치 않는 재렌더링이 발생하거나 의존성 누락으로 버그가 생길 수 있습니다.</li>
</ul>
</li>
</ol>
<h2>props Drilling</h2>
<p>Q. props Drilling을 설명하고, 이를 해결하는 방법을 설명하시오.</p>
<ul>
<li>설명: Props Drilling이란 데이터를 하위 컴포넌트에 전달하기 위해 중간 컴포넌트를 거쳐 props를 계속 전달해야 하는 상황을 의미합니다. 중간 컴포넌트가 데이터를 사용하지 않더라도 전달 역할을 하게 됩니다.</li>
<li>해결 방법:
<ol>
<li><strong>Context API</strong>: 데이터를 전역적으로 공유할 수 있는 구조로 만들어 props Drilling을 방지합니다.</li>
<li><strong>상태 관리 라이브러리</strong>: Redux, MobX, Zustand 등을 사용해 중앙 상태에서 데이터를 관리합니다.</li>
<li><strong>컴포넌트 구조 개선</strong>: 컴포넌트를 재구조화하여 Drilling을 최소화합니다.</li>
</ol>
</li>
</ul>
<h2>Context API</h2>
<p>Q. 리액트에서 Context API를 사용할 경우, 성능에 문제가 되는 경우는 어떤 때인가? 해결방법은 무엇인가?</p>
<ul>
<li>문제 발생 상황: Context API는 값을 소비하는 모든 하위 컴포넌트를 리렌더링합니다. 컨텍스트 값이 빈번히 변경될 경우, 불필요한 렌더링이 발생할 수 있다.</li>
<li>해결 방법:
<ol>
<li>컨텍스트 분리: 컨텍스트를 여러 개로 나눠 필요한 부분만 리렌더링되도록 분리</li>
<li>메모이제이션: React.memo와 useMemo를 활용해 컴포넌트를 최적화</li>
<li>사용자 정의 Hook: 컨텍스트 값과 비즈니스 로직을 캡슐화하여 필요한 컴포넌트만 업데이트</li>
</ol>
</li>
</ul>
<h2>useEffect vs. useLayoutEffect</h2>
<p>Q. 리액트에서 사용되는 useEffect와 useLayoutEffect를 설명하시오.</p>
<ul>
<li>useEffect:
<ul>
<li>비동기 작업(예: 데이터 패칭, 구독 설정 등)을 수행합니다.</li>
<li><strong>DOM이 그려진 뒤 비동기적으로 실행</strong>됩니다. 사용자에게 렌더링이 먼저 보여진 뒤 작업이 수행됩니다.</li>
</ul>
</li>
<li>useLayoutEffect:
<ul>
<li>DOM이 업데이트된 직후, <strong>브라우저가 그리기 전에 동기적으로 실행</strong>됩니다.</li>
<li>레이아웃 측정을 해야 하거나 DOM 변경이 필요한 경우 사용됩니다.</li>
</ul>
</li>
</ul>
<h2>key</h2>
<p>Q. 재조정 알고리즘 배열에서 key를 명시하는 이유를 설명하시오.</p>
<ul>
<li>이유: React는 DOM을 효율적으로 업데이트하기 위해 Diffing 알고리즘을 사용한다. key는 컴포넌트가 고유한 식별자로, React가 어떤 항목이 변경, 추가, 삭제되었는지 효율적으로 판단하게 한다.</li>
<li>문제 발생: key를 적절히 설정하지 않으면, 불필요한 DOM 업데이트가 발생하거나 렌더링 순서가 꼬일 수 있습니다.</li>
</ul>
<h2>SSR vs. CSR</h2>
<p>Q. SSR과 CSR을 설명하고 둘의 차이점을 설명하시오.</p>
<ul>
<li>SSR (Server-Side Rendering): 서버에서 HTML을 렌더링한 뒤 클라이언트에 전달합니다</li>
<li>CSR (Client-Side Rendering): 브라우저에서 자바스크립트가 HTML을 렌더링합니다.</li>
<li>차이점:
<ul>
<li>SSR: 빠른 초기 렌더링, SEO 친화적, 서버 부담 증가.</li>
<li>CSR: 서버 부담 적음, 초기 로딩 느림, 사용자 인터랙션 이후 빠른 업데이트.</li>
</ul>
</li>
</ul>
<h2>critical rendering path</h2>
<p>Q. 브라우저 렌더링에서 critical rendering path을 설명하시오.</p>
<ul>
<li>설명: Critical Rendering Path는 HTML, CSS, 자바스크립트를 브라우저가 화면에 렌더링하기 위해 처리하는 과정을 뜻합니다.</li>
<li>과정:
<ol>
<li>HTML 파싱 → DOM 생성</li>
<li>CSS 파싱 → CSSOM 생성</li>
<li>DOM + CSSOM → Render Tree 생성</li>
<li>Layout → Paint → Composite</li>
</ol>
</li>
</ul>

<li>예시:
<pre><code class="language-js">element.style.width = "100px";  // DOM 변경

</code></pre>
</li>
</ul>

<h2>CORS preflight</h2>
<p>Q. CORS preflight있는 이유와 preflight가 발생하지 않는 조건을 설명하시오.</p>
<ul>
<li>이유: 서버에 안전하지 않은 요청(method: PUT, DELETE, PATCH 등)을 보내기 전, 브라우저가 서버의 허가를 받기 위해 예비 요청을 보냅니다.</li>
<li>발생하지 않는 조건:
<ul>
<li>메서드: GET, POST, HEAD</li>
<li>헤더: 간단한 헤더만 포함 (예: Content-Type: application/json)</li>
</ul>
</li>
</ul>

</ul>
