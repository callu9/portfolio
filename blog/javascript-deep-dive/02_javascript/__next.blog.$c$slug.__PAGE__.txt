<h2>2.2. 자바스크립트의 표준화</h2>
<p>1996년 8월, 마이크로소프트는 자바스크립트의 파생 버전인 “JScript”를 인터넷 익스플로러 3.0에 탑재했고, 자바스크립트가 표준화되지 못하고 적당히 호환되었다는 문제가 발생했다. 즉, 넷스케이프 커뮤니케이션즈와 마이크로소프트는 자사 브라우저의 시장 점유율을 높이기 위해 자사 브라우저에서만 동작하는 기능을 경쟁적으로 추가하기 시작했다는 것.</p>
<p>이로 인해 브라우저에 따라 웹페이지가 정상적으로 동작하지 않는 <strong>크로스 브라우징 이슈</strong>가 발생하기 시작했고, 결과적으로 모든 브라우저에서 정상적으로 동작하는 웹페이지를 개발하기 위하여 자바스크립트의 파편화를 방지하고 모든 브라우저에서 정상적으로 동작하는 표준화된 자바스크립트의 필요성이 대두되기 시작했다. 이를 위해 1996년 11월, 넷스케이프 커뮤니케이션즈는 컴퓨터 시스템의 표준을 관리하는 비영리 표준화 기구인 ECMA 인터내셔널에 자바스크립트의 표준화를 요청한다.</p>

<h2>2.3. 자바스크립트 성장의 역사</h2>
<p>초창기 자바스크립트는 웹페이지의 보조적인 기능을 수행하기 위해 한정적인 용도로 사용되었다.
이 시기에 대부분의 로직은 주로 웹 서버에서 실행되었고, 브라우저는 서버로부터 전달받은 HTML과 CSS를 단순히 <em>렌더링</em>하는 수준이었다.</p>
<ul>
<li>렌더링(rendering): HTML, CSS, 자바스크립트로 작성된 문서를 해석해서 브라우저에 시각적으로 출력하는 것</li>
</ul>
<h3>2.3.1. Ajax</h3>
<h4>1) Ajax의 등장</h4>
<p>1999년, XHMLHttpRequest라는 이름으로 자바스크립트를 이용해 <strong>서버와 브라우저가 비동기(asynchronous) 방식으로 데이터를 교환</strong>할 수 있는 통신 기능인 Ajax(Asynchronous Javascript and XML)가 등장했다.</p>
<h4>2) Ajax의 장점</h4>
<p>이전의 웹페이지와 같이 웹페이지에서 변경할 필요가 없는 부분을 렌더링 하지 않고, 서버로부터 필요한 데이터만 전송받아 변경해야 하는 부분만 한정적으로 렌더링하는 방식이 가능해졌다.
즉, 웹 브라우저에서도 데스크톱 애플리케이션과 유사한 빠른 성능과 부드러운 화면 전환이 가능해졌다. (처음부터 다시 렌더링하여 화면이 전환되면 순간적으로 화면 깜빡임 현상이 발생하는 한계가 사라졌다.)</p>
<h4>3) Ajax 사용 사례</h4>
<p>2005년, 구글 맵스(Google Maps)는 웹 브라우저에서 자바스크립트와 Ajax를 기반으로 동작하여, 데스크톱 애플리케이션과 비교했을 때 손색이 없을 정도의 성능과 부드러운 화면 전환 효과를 보여주었다.</p>
<h3>2.3.2. jQuery</h3>
<h4>1) jQuery의 등장</h4>
<p>2006년, jQuery의 등장으로 다소 번거롭고 논란이 있던 DOM(Document Object Model)을 더욱 쉽게 제어할 수 있게 되었고 크로스 브라우징 이슈도 어느정도 해결되었다.</p>
<h4>2) jQuery의 장점</h4>
<p>자바스크립트보다 배우기 쉽고 직관적이며, DOM 제어가 쉽다는 장점이 있다.</p>
<h3>2.3.3. V8 자바스크립트 엔진</h3>
<h4>1) V8 자바스크립트 엔진의 등장</h4>
<p>2008년, 구글의 자바스크립트 엔진은 자바스크립트로 웹 애플리케이션을 구축하려는 시도가 늘면서 더욱 빠르게 동작하는 자바스크립트 엔진의 필요성에 대한 요구를 충족하기 위해 등장하였다.
V8 자바스크립트 엔진의 등장으로 자바스크립트는 데스크톱 애플리케이션과 유사한 사용자 경험(UX, User Experience)을 제공할 수 있는 웹 애플리케이션 프로그래밍 언어로 정착하게 되었다.</p>
<h4>2) V8 자바스크립트 엔진의 장점</h4>
<p>빠른 성능 및 과거 웹 서버에서 수행되던 로직을 클라이언트(브라우저)로 이동</p>
<h3>2.3.4. Node.js</h3>
<h4>1) Node.js의 등장</h4>
<p>2009년, 라이언 달(Ryan Dhal)이 구글 V8 자바스크립트 엔진으로 빌드된 자바스크립트 런타임 환경(runtime environment)인 Node.js를 발표하였다.
Node.js는 자바스크립트를 브라우저 이외의 환경에서도 동작할 수 있도록 자바스크립트 엔진을 브라우저에서 독립시킨 자바스크립트 실행 환경이다.
주로 서버 사이드 애플리케이션 개발에 주로 사용되며, 이에 필요한 모듈, 파일 시스템, HTTP 등 빌트인 API를 제공한다.
자바스크립트는 <strong>크로스 플랫폼</strong>을 위한 가장 중요한 언어로 주목받게 되었다.</p>
<h4>2) Node.js의 장점</h4>
<ul>
<li>서버 사이드 어플리케이션 개발 가능: 자바스크립트 엔진을 기반으로 하므로, 프론트엔드와 백엔드 영역에서 자바스크립트를 사용할 수 있다는 동형성(isomorphic)으로 인해 별도의 언어 학습 시간 감축 가능.</li>
<li>요청 처리 성능이 좋음: 비동기 I/O를 지원하며, 단일 스레드 이벤트 루프 기반으로 동작하기 때문이다. Node.js는 데이터를 실시간으로 처리하기 위해 I/O가 빈번하게 발생하는 SPA에 적합하다. (하지만 CPU 사용률이 높은 애플리케이션에는 권장하지 않음.)</li>
</ul>
<h3>2.3.5. SPA 프레임워크</h3>
<h4>1) SPA 프레임워크의 등장</h4>
<p>개발 규모와 복잡도가 상승함으로써, 변경에 유연하면서 확장하기 쉬운 애플리케이션 아키텍처 구축을 위해 CBD(Component based development) 방법론을 기반으로 SPA(Single Page Application)가 대중화되었다. Angular, React, Vue.js, Svelte 등 다양한 SPA 프레임워크/라이브러리 또한 많은 사용층을 확보하게 되었다.</p>
<h2>2.4. 자바스크립트와 ECMAScript</h2>
<h3>ECMAScript</h3>
<p>자바스크립트의 표준 사양인 ECMA-262를 말하며, 프로그래밍 언어의 값, 타입, 객체와 프로퍼티, 함수, 표준 빌트인 객체(standard built-in object) 등 핵심 문법을 규정한다.</p>
<p>자바스크립트란, 기본 뼈대인 ECMAScript와 브라우저가 별도 지원하는 <strong>클라이언트 사이드 Web API</strong>를 아우르는 개념.</p>
<ul>
<li>클라이언트 사이드 Web API
<ul>
<li>DOM, BOM, Canvas, XMLHttpRequest, fetch, requestAnimationFrame, SVG, Web Storage, Web Component, Web Worker... 등</li>
<li>ECMAScript와는 별도로 월드 와이드 웹 콘소시엄(World Wide Consortium:W3C)에서 별도의 사양으로 관리</li>
<li>참고: <a href="https://developer.mozilla.org/ko/docs/Web/API">MDN web docs - Web API</a></li>
</ul>
</li>
</ul>
<h2>2.5. 자바스크립트의 특징</h2>
<ul>
<li>HTML, CSS와 함께 웹을 구성하는 요소 중 하나로 <strong>웹 브라우저에서 동작하는 유일한 프로그래밍 언어</strong>이며, 별도의 컴파일 작업을 수행하지 않는 <strong>인터프리터 언어</strong> (interpreter language)</li>
<li>모던 자바스크립트 엔진은 인터프리터와 컴파일러의 장점을 결합해 일부 소스코드를 컴파일하고 실행하여, 인터프리터 언어의 장점인 동적 기능 지원을 유지하고 실행 속도가 느리다는 단점을 해결
<ul>
<li>크롬 V8, 파이어폭스 SpiderMonkey, 사파리 JavaScriptCore, 마이크로소프트 엣지 Chakra 등</li>
</ul>
</li>
<li>자바스크립트는 명령형, 함수형, <strong>프로토타입 기반 객체지향</strong> 프로그래밍을 지원하는 멀티 패러다임 언어</li>
</ul>
<h3>인터프리터 언어 vs. 컴파일러 언어</h3>
<p>|     | 컴파일러 언어 | 인터프리터 언어 |
| --- | --- | --- |
| <strong>변환 및 실행</strong> | 코드 실행 전 단계인 컴파일 타임에 소스코드 전체를 머신코드로 변환 후 실행 | 코드 실행 단계인 런타임에 문 단위로 한 줄씩 바이트코드로 변환 후 실행 |
| <strong>실행 파일 생성</strong> |    O    |    X    |
| <strong>실행 단계 분리</strong> |    O    |    X    |
| <strong>수행 횟수</strong>      |   1번   | 코드 실행될 때마다 반복 수행 |
| <strong>코드 실행 속도</strong> |  빠르다  |  비교적 느리다 |</p>
